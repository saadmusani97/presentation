Foundations

What is OOP?

Imagine you're building a game with 10 dogs. Without OOP, you'd need 10 separate variables for each dog's name, breed, and age. With OOP, you write a Dog blueprint once, then create as many dogs as you want from it!

Object-Oriented Programming (OOP) is a way of writing code by grouping related data and actions together into "objects". Think of objects like real-world things — a Dog has a name, a breed, and can bark.

01
Class
The blueprint / recipe
02
Object
A real thing made from the blueprint
03
Attribute
Data stored inside an object
04
Method
An action the object can do
Python
# Without OOP — messy!
dog1_name  = "Rex"
dog1_breed = "Labrador"
dog2_name  = "Buddy"
dog2_breed = "Poodle"

# With OOP — clean! ✨
class Dog:
    def __init__(self, name, breed):
        self.name  = name   # attribute
        self.breed = breed  # attribute

    def bark(self):         # method
        return f"{self.name} says: Woof! 🐕"

# Create dogs from the blueprint
dog1 = Dog("Rex", "Labrador")
dog2 = Dog("Buddy", "Poodle")

print(dog1.bark()) # Rex says: Woof! 🐕
print(dog2.bark()) # Buddy says: Woof! 🐕

Constructor

The __init__ Method

Python
class Student:
    def __init__(self, name, marks):
        # Runs automatically when created!
        self.name  = name
        self.marks = marks

    def grade(self):
        if self.marks >= 90:
            return "A+ 🌟"
        elif self.marks >= 70:
            return "B 👍"
        else:
            return "C — keep studying! 📚"

    def greet(self):
        return f"Hi! I'm {self.name}."

# __init__ runs automatically here ↓
s1 = Student("Alice", 95)
s2 = Student("Bob", 72)

print(s1.greet())  # Hi! I'm Alice.
print(s1.grade())  # A+ 🌟
print(s2.grade())  # B 👍

__init__ is the "setup" method. It runs automatically the moment you create a new object. You never call it yourself — Python calls it for you!

It's like filling out a form when you sign up for a new account — it happens right away, once, at the start.

  • ①

    self = this specific object

    Always the first parameter. Python passes it automatically. You never send it yourself.

  • ②

    Set your data here

    self.name = name saves the name inside this object.

  • ③

    One-time setup only

    It runs once when creating. For all other actions, write separate methods.

Inheritance

Children Learn from Parents

Inheritance lets a child class get all the methods and data from a parent class for free. A Dog IS-AN Animal, so it can breathe, eat, and sleep — without you writing those methods again.

🐾 Animal
🐶 Dog
🐱 Cat
🐦 Bird
Key Idea

Write common code once in the parent. All child classes get it automatically. Add only what's unique to each child.

Python
class Animal:         # Parent class
    def __init__(self, name):
        self.name = name

    def eat(self):
        return f"{self.name} is eating 🍽️"

    def sleep(self):
        return f"{self.name} is sleeping 💤"

# Dog INHERITS from Animal
class Dog(Animal):
    def bark(self):    # Dog's own method
        return f"{self.name} says: Woof! 🐕"

rex = Dog("Rex")

# Inherited from Animal
print(rex.eat())    # Rex is eating 🍽️
print(rex.sleep())  # Rex is sleeping 💤
# Dog's own method
print(rex.bark())   # Rex says: Woof! 🐕

super()

Calling Parent's Methods

Python
class Animal:
    def __init__(self, name):
        self.name = name

    def introduce(self):
        return f"I am {self.name}"

class Dog(Animal):
    def __init__(self, name, breed):
        # Call parent's __init__ first
        super().__init__(name)
        self.breed = breed  # add extra

    def introduce(self):
        # Get parent's intro, then add more
        base = super().introduce()
        return f"{base}, a {self.breed} 🐕"

d = Dog("Rex", "Labrador")
print(d.introduce())
# I am Rex, a Labrador 🐕

super() is how a child class talks to its parent. Instead of repeating the parent's code, you call super() to run it and then add your own extra steps.

Think of it like: "Do what Mom does, then also do my own thing."

  • 01

    Always call super().__init__ first

    This sets up everything the parent needs before you add new stuff.

  • 02

    Extend, don't replace

    Use super().method() then add extra logic on top.

  • 03

    No hardcoding parent name

    If you rename the parent class, super() still works automatically.

Inheritance Types

Five Types of Inheritance

Python supports five forms of inheritance. Don't worry — you'll use Single most of the time. The others are good to know as you grow.

01

Single

One parent → one child.
class Dog(Animal)
Most common. Start here.

02

Multiple

Two parents → one child.
class Duck(Bird, Swimmer)
A duck flies AND swims!

03

Multilevel

Chain: Grandparent → Parent → Child. Like: Animal → Dog → Puppy.

04

Hierarchical

One parent → many children. Animal → Dog, Cat, Bird, Fish.

05

Hybrid

A mix of the above types. Python handles name conflicts with the MRO (Method Resolution Order) — a smart lookup chain so each class is checked only once.

Polymorphism

Same Word, Different Actions

Python
class Dog:
    def speak(self):
        return "Woof! 🐶"

class Cat:
    def speak(self):
        return "Meow! 🐱"

class Bird:
    def speak(self):
        return "Tweet! 🐦"

    print(pet.speak())

# Meow! 🐱
# Tweet! 🐦

Polymorphism means "many forms". The same method name (speak) behaves differently depending on which object calls it.

You write one loop that calls speak() — Python figures out which version to run at the right moment.

Method Overriding

Rewriting a Parent's Method

Method Overriding is when a child class writes its own version of a method that already exists in the parent. Python always uses the child's version for child objects.

The parent's version is still there — you can access it with super() if you need it.

  • !

    Name must match exactly

    A typo creates a brand new method instead. Python won't warn you.

  • !

    Most specific wins

    Python always calls the child's version first. The parent is used only as fallback.

Output
Some animal sound...
Rex says: Woof! 🐶
Luna says: Meow! 🐱
Tweety says: Tweet! 🐦
Python
class Animal:
    def __init__(self, name):
        self.name = name

class Dog(Animal):

Duck Typing

If It Can Do It, Use It!

"If it walks like a duck and quacks like a duck — it's a duck." 🦆

— Python's Golden Rule

Python doesn't ask "what type is this object?" — it asks "does this object have the method I need?"

No shared parent class needed. If the object has a .speak() method, Python is happy to call it — doesn't matter where it came from.

Real life analogy

A remote control works with any TV brand — Samsung, Sony, LG. It doesn't care which brand it is, only that it can change channels.

Python
# No shared parent — totally different classes!
class Dog:
    def speak(self):
        return "Woof! 🐶"

class Person:
    def speak(self):
        return "Hello! 👋"

class Robot:
    def speak(self):
        return "BEEP BOOP 🤖"

# Python only checks: "does it have speak()?"
def make_speak(thing):
    print(thing.speak())

for obj in [Dog(), Person(), Robot()]:
    make_speak(obj)

# Woof! 🐶
# Hello! 👋
# BEEP BOOP 🤖

Polymorphism in Action

Write One Function, Use Anywhere

The true magic of polymorphism: write a function once, and it works with any object that has the right methods. Even objects you haven't created yet! Python's built-in len() does exactly this.

Python
class Circle:
    def __init__(self, r):
        self.r = r
    def area(self):
        return 3.14 * self.r * self.r

class Square:
    def __init__(self, s):
        self.s = s
    def area(self):
        return self.s * self.s

class Rectangle:
    def __init__(self, w, h):
        self.w, self.h = w, h
    def area(self):
        return self.w * self.h

# One function — works for ALL shapes!
def print_area(shape):
    print(f"Area = {shape.area()}")

shapes = [Circle(5), Square(4), Rectangle(3,6)]
for s in shapes:
    print_area(s)
Output
Area = 78.5
Area = 16
Area = 18
Python Built-ins use Polymorphism too!

len("hello") → 5
len([1, 2, 3]) → 3
len({"a": 1}) → 1

One function. Three different types. Polymorphism is everywhere in Python!

🚀 Add a Triangle later

Just create a Triangle class with an area() method and put it in the list. The print_area() function needs zero changes.

Make + Work Your Way

What does 2 + 3 give? 5. What about "Hi" + "!"? "Hi!". The + operator does different things depending on the type.

You can define what +, ==, and print() do for your own classes using special dunder methods (double-underscore methods).

Python
class ShoppingBag:
    def __init__(self, items):
        self.items = items

    # Combine two bags with +
    def __add__(self, other):
        combined = self.items + other.items
        return ShoppingBag(combined)

    # How many items?
    def __len__(self):
        return len(self.items)

    # Friendly print
    def __str__(self):
        return f"Bag: {self.items}"

bag1 = ShoppingBag(["apple", "milk"])
bag2 = ShoppingBag(["bread"])

big_bag = bag1 + bag2  # uses __add__

print(big_bag)        # Bag: ['apple','milk','bread']
print(len(big_bag))   # 3

Abstract Classes

Making Rules Everyone Must Follow

An Abstract Class is like a job contract — it says "everyone hired must be able to do these tasks". You can't hire the contract itself (can't create the abstract class), but every person you hire (subclass) must fulfil it.

Use @abstractmethod to mark methods that subclasses must write. Python will throw an error if they don't.

When to Use

When you have a group of classes that should all have the same methods, but each one does it differently — like different types of employees all needing a work() method.

Python
from abc import ABC, abstractmethod

class Employee(ABC):  # Contract!
    @abstractmethod
    def work(self):    # Must override!
        pass

    @abstractmethod
    def salary(self):  # Must override!
        pass

    def greet(self):   # Normal method ✅
        return "Hi, I work here!"

class Designer(Employee):
    def work(self):
        return "Making beautiful UI 🎨"
    def salary(self):
        return 50000

d = Designer()
print(d.work())   # Making beautiful UI 🎨
print(d.greet())  # Hi, I work here!
# Employee()  ← TypeError: Can't create!

Type Checking

isinstance() & issubclass()

isinstance(obj, Class) asks: "Is this object a member of this class (or any of its children)?" It's like checking if someone holds a passport from a country.

issubclass(Child, Parent) asks: "Does this class inherit from that class?" Checks the family tree, not a specific object.

Tip

Prefer duck typing normally. Use isinstance only when you really need to treat different types differently — like in a form validator.

Python
class Animal: pass
class Dog(Animal): pass
class Cat(Animal): pass

rex  = Dog()
luna = Cat()

# isinstance — check specific object
print(isinstance(rex, Dog))    # True
print(isinstance(rex, Animal)) # True — it's a Dog which IS an Animal
print(isinstance(rex, Cat))    # False

# issubclass — check class family
print(issubclass(Dog, Animal)) # True
print(issubclass(Cat, Dog))   # False

# Practical use: give different greetings
def greet(pet):
    if isinstance(pet, Dog):
        print("🐶 Good boy!")
    elif isinstance(pet, Cat):
        print("🐱 Hello kitty!")

greet(rex)   # 🐶 Good boy!
greet(luna)  # 🐱 Hello kitty!

MRO

Which Version Gets Called?

When two parent classes have the same method, Python needs to decide whose version to use. The Method Resolution Order (MRO) is the lookup list Python follows — always left to right.

Think of it like a queue: Python checks each class in order and calls the first match it finds.

A (grandparent)
B (parent)
C (parent)
D (child)
MRO: D → B → C → A → object
Python
class A:
    def hello(self): return "Hello from A"

class B(A):
    def hello(self): return "Hello from B"

class C(A):
    def hello(self): return "Hello from C"

# D inherits from BOTH B and C
class D(B, C):
    pass

d = D()
print(d.hello())  # "Hello from B" — B comes first

# Check the order Python will look
print(D.mro())
# [D, B, C, A, object]

# Python checks in that order:
# D? No hello. → B? Yes! Uses B's hello.

from abc import ABC, abstractmethod

class Pet(ABC):
    def __init__(self, name):
        self.name = name

    @abstractmethod
    def sound(self): pass

    @abstractmethod
    def food(self): pass

    def profile(self):
        return (
            f"Name: {self.name}\n"
            f"Sound: {self.sound()}\n"
            f"Eats: {self.food()}"

    def food(self):  return "Fish 🐟"

store = [Dog("Rex"), Cat("Luna")]

    print("---")

Encapsulation

Keeping Data Safe

Encapsulation means keeping data protected inside a class. Python uses naming conventions (not strict rules) to signal how private something should be.

Think of a ATM machine — you can press buttons to get money, but you can't reach inside and take cash directly!

  • ○

    name — Public 🟢

    Anyone can read and change it. Default behaviour.

  • ○

    _name — Protected 🟡

    Convention says: only use inside the class or child classes. Not enforced.

  • ○

    __name — Private 🔴

    Python renames it internally. Harder (not impossible) to access from outside.

Python
class Piggybank:
    def __init__(self, owner):
        self.owner    = owner  # public 🟢
        self._color   = "pink" # protected 🟡
        self.__coins  = 0      # private 🔴

    def add(self, amount):
        if amount > 0:
            self.__coins += amount
            print(f"Added ₹{amount} 🪙")

    def balance(self):     # safe way to read
        return self.__coins

bank = Piggybank("Alice")
bank.add(50)
bank.add(30)
print(bank.balance())    # 80
print(bank.owner)        # Alice
# bank.__coins  ← AttributeError! 🔴

Pitfalls

Common Beginner Mistakes

Mistake 01

Forgetting self. Every method in a class must have self as the first parameter. If you forget it, Python passes the object and you'll get a confusing TypeError.

Mistake 02

Forgetting super().__init__(). If you write __init__ in a child class without calling super(), the parent's attributes won't be set up — and you'll get AttributeErrors later.

Mistake 03

Typo in overridden method name. Writing speeek() instead of speak() creates a new method silently — it does NOT override the parent. Very hard to debug!

Mistake 04

Mutable class attribute as default. A list set as a class attribute is shared by ALL instances — changing it in one object affects every other object!

Mistake 05

Too many levels of inheritance. Beyond 2–3 levels, code becomes hard to follow. Prefer simple, shallow hierarchies. If it's getting complex, use composition instead.

Mistake 06

Not calling the parent's version. When overriding, you might accidentally lose the parent's useful behaviour. Always check: should I call super().method() too?

Best Practices

Simple OOP Rules to Follow

🎯

One Job Per Class

Each class should do one thing well. A Dog class handles dog stuff — not also sending emails!

🔁

Don't Repeat Yourself

If two classes share the same code, move it to a parent class. Write it once, inherit everywhere.

🔌

Open for Growth

Add new features by adding new classes — not by editing existing working code.

🦆

Use Duck Typing

Don't use isinstance() everywhere. Write functions that work with anything that has the right method.

📝

Name Things Clearly

Good names are self-documenting. Dog.bark() is obvious. D.b() is not. Future you will thank present you.

🧱

Keep It Shallow

Limit inheritance to 2–3 levels. Deep chains make code confusing. Prefer "has-a" (composition) over deep "is-a".

Dunder Methods

__str__ — Make print() Friendly

By default, print(my_dog) shows something ugly like <__main__.Dog object at 0x7f...>. By defining __str__, you control exactly what gets printed — clean and readable!

__repr__ is similar but for developers — shown in the Python shell/REPL. A good __repr__ looks like valid Python code to recreate the object.

Quick Rule

Always define at least __str__ for user-facing output. Define __repr__ too if you want clean REPL output.

Python
class Dog:
    def __init__(self, name, breed, age):
        self.name  = name
        self.breed = breed
        self.age   = age

    # What users see with print()
    def __str__(self):
        return (
            f"🐶 {self.name} the {self.breed}"
            f" ({self.age} years old)"
        )

    # What developers see in the REPL
    def __repr__(self):
        return (
            f"Dog('{self.name}',"
            f"'{self.breed}', {self.age})"
        )

rex = Dog("Rex", "Labrador", 3)

# Without __str__: <Dog object at 0x7f...>
# With __str__:
print(rex)
# 🐶 Rex the Labrador (3 years old)

print(repr(rex))
# Dog('Rex', 'Labrador', 3)

Key Takeaways

🏗️
Classes are Blueprints
Write a class once, create unlimited objects from it — each with its own data.
🔗
Inheritance Saves Time
Child classes get all parent methods for free. Only write what's unique to each child.
🎭
Polymorphism = Flexibility
Write one function that works with many types. Add new types without changing existing code.
🦆
Duck Typing is Pythonic
Python cares about what an object can do, not what it is. Focus on methods, not types.
📜
Abstract Classes Enforce Rules
Use ABC when all child classes must implement the same methods — it documents and enforces your design.
✨
Dunders Make Magic
__str__, __add__, __len__ let your classes work naturally with Python's built-in syntax.
